# Статус проекта и план разработки

## Общая цель проекта

**Генератор и инкрементальный обновитель SCADA-проектов** для системы **Monitor/PTUSA** из исходных документов:
- **Вход**: PDF-описания проекта + Lua-скрипты логики
- **Выход**: XML-файлы SCADA-проекта Monitor (`Units.xml`, `Project.xml`, и т.д.)
- **Особенность**: не пересобирает проект с нуля, а **инкрементально обновляет** существующий при изменениях PDF/Lua

---

## Что уже сделано

### 1. Архитектура проекта (Clean Architecture)

Проект структурирован по принципам чистой архитектуры:

- **`domain`** — доменная модель (бизнес-сущности):
  - `ProjectModel`, `ProjectDescription` — модели проекта
  - `Equipment`, `EquipmentType` — оборудование
  - `Tag`, `TagType` — теги/сигналы SCADA
  - `ControlLogic` — алгоритмическая логика (Lua)

- **`usecase`** — бизнес-логика (use cases):
  - `ScadaProjectService` — фасад для операций над проектом
  - `PdfParser`, `LuaParser` — интерфейсы парсеров
  - `ProjectGenerator`, `DeltaUpdater` — интерфейсы генерации/обновления
  - `ProjectModelAssembler` — сборщик доменной модели

- **`adapter`** — адаптеры к внешним библиотекам:
  - `PdfBoxPdfParser` — парсер PDF на базе Apache PDFBox 3.x
  - `SimpleLuaParser` — простой парсер Lua-файлов

- **`generator`** — генераторы выходных файлов:
  - `XmlProjectGenerator` — генератор XML (пока абстрактный формат)
  - `NoOpDeltaUpdater` — заглушка для инкрементального обновления

- **`cli`** — командный интерфейс:
  - `ScadaGeneratorCli` — точка входа
  - `GenerateCommand`, `UpdateCommand`, `WatchCommand` — команды CLI

### 2. Рабочий MVP (минимально жизнеспособный продукт)

**Что работает сейчас:**

- ✅ CLI-приложение собирается и запускается (`java -jar ...`)
- ✅ Команда `generate` принимает PDF + Lua и генерирует базовый `project.xml`
- ✅ Парсер PDF читает файл и извлекает сырой текст (MVP)
- ✅ Парсер Lua читает файл целиком и создаёт один блок `ControlLogic`
- ✅ Генератор создаёт XML с оборудованием, тегами и Lua-кодом
- ✅ Тесты покрывают ключевые компоненты (парсер Lua, сборщик модели, генератор XML)

**Что пока не работает полноценно:**

- ⚠️ PDF-парсер не извлекает структурированные данные (оборудование, теги, параметры)
- ⚠️ Lua-парсер не анализирует код, не находит используемые теги (`DI/DO/AI`)
- ⚠️ Генератор создаёт абстрактный XML, а не формат Monitor
- ⚠️ Инкрементальное обновление (`update`) — заглушка, кидает `UnsupportedOperationException`
- ⚠️ Команда `watch` — заглушка

### 3. Инфраструктура разработки

- ✅ **Maven Wrapper** (`mvnw`, `mvnw.cmd`) — сборка без установленного Maven
- ✅ **Зависимости**: Picocli (CLI), PDFBox 3.0.2 (PDF), JUnit 5 (тесты)
- ✅ **Сборка jar с зависимостями** — работает `java -jar` из коробки
- ✅ **Тесты** — JUnit 5, прогоняются через `mvnw test`

  - `main.plua`, `prg.lua`, `alpmagate.lua` и др. — Lua-логика

## Как это сделано (технические решения)

### Архитектурный подход

Проект следует принципам **Clean Architecture**:
- **Независимость от фреймворков** — только стандартные библиотеки Java + Maven
- **Тестируемость** — бизнес-логика изолирована от адаптеров
- **Расширяемость** — новые парсеры/генераторы добавляются через интерфейсы

### Поток данных (текущий MVP)

```
PDF файл → PdfBoxPdfParser → ProjectDescription (только projectId, пустые equipments)
Lua файл → SimpleLuaParser → List<ControlLogic> (один блок с целым файлом)
         ↓
ProjectModelAssembler → ProjectModel (объединяет описание + логику)
         ↓
XmlProjectGenerator → project.xml (абстрактный формат)
```

### Технологический стек

- **Java 17** — язык и платформа
- **Maven** — система сборки
- **Apache PDFBox 3.0.2** — парсинг PDF
- **Picocli 4.7.5** — CLI-фреймворк
- **JUnit 5** — тестирование
- **JDK DOM API** — генерация XML (стандартная библиотека)

## Что ещё нужно сделать

### Этап 1: Генератор под формат Monitor (на основе "Моцареллы")

**Цель**: научить генератор создавать реальные файлы Monitor, похожие на `O1_MOZARELLA.files/*.xml`

**Задачи:**

1. **Анализ эталонного проекта "Моцарелла"**
   - Разобрать структуру `Units.xml` (юниты, свойства, каналы)
   - Разобрать структуру `Project.xml` (карты, контейнеры)
   - Зафиксировать маппинг: наши `Equipment`/`Tag` → Monitor XML

2. **Расширение доменной модели**
   - Добавить поля в `Equipment` для Monitor-специфичных данных (`typename`, `framepos`, `isdynamic`)
   - Добавить поля в `Tag` для Monitor-свойств (`channelid`, `propmodel`, `proptype`, `caption`)
   - Возможно, новые классы для Monitor-юнитов/свойств

3. **Генератор Monitor XML**
   - Переписать `XmlProjectGenerator` для генерации `Units.xml` в формате Monitor
   - Добавить генерацию `Project.xml` (базовая структура)
   - Тесты, сравнивающие результат с эталонными файлами "Моцареллы"

**Ожидаемый результат**: команда `generate` создаёт `Units.xml` и `Project.xml`, которые можно открыть в Monitor

### Этап 2: Умный парсинг Lua

**Цель**: извлекать из Lua реальные теги и связывать их с оборудованием

**Задачи:**

1. **Парсер Lua-вызовов**
   - Находить вызовы `DI("...")`, `DO("...")`, `AI("...")`, `AO("...")`
   - Извлекать имена тегов (например, `"ALPMAGW5DI480503"`)
   - Заполнять `ControlLogic.tagIds`

2. **Анализ структуры Lua**
   - Разбивать большой файл на логические блоки (функции, объекты)
   - Создавать несколько `ControlLogic` вместо одного большого блока

3. **Связывание тегов с оборудованием**
   - По шаблонам имён (`ALPMAGW5DI...`) определять, к какому `Equipment` относится тег
   - Автоматически создавать `Tag` в нужном `Equipment`

**Ожидаемый результат**: из `main.plua` "Моцареллы" извлекаются все используемые теги и связываются с юнитами

### Этап 3: Структурированный парсинг PDF

**Цель**: извлекать из PDF реальное оборудование, агрегаты, параметры

**Задачи:**

1. **Парсинг таблиц PDF**
   - Использовать PDFBox для извлечения таблиц
   - Распознавать структуру: агрегаты, линии, оборудование, параметры

2. **Извлечение структуры проекта**
   - Находить заголовки разделов (агрегаты, линии)
   - Извлекать списки оборудования с параметрами
   - Создавать `Equipment` с правильными `id`, `name`, `type`

3. **Связывание PDF и Lua**
   - Сопоставлять оборудование из PDF с тегами из Lua
   - Заполнять `Tag` данными из обоих источников

**Ожидаемый результат**: из `O1-Моцарелла ЧБ.pdf` извлекается список оборудования, который попадает в `Units.xml`

### Этап 4: Инкрементальное обновление (DeltaUpdater)

**Цель**: обновлять существующий Monitor-проект без полной пересборки

**Задачи:**

1. **Хранение метаданных**
   - Сохранять предыдущую версию `ProjectModel` в `metadataStore`
   - Хранить информацию о том, что уже выгружено в Monitor

2. **Сравнение моделей**
   - Сравнивать `oldModel` и `newModel` на уровне домена
   - Определять: что добавлено, что изменено, что удалено

3. **Реализация DeltaUpdater**
   - Обновлять только изменённые части `Units.xml`/`Project.xml`
   - Не трогать файлы, которые не изменились
   - Сохранять ручные правки инженеров (где возможно)

**Ожидаемый результат**: команда `update` реально обновляет проект, а не пересобирает с нуля

### Этап 5: Команда watch (опционально)

**Цель**: автоматически отслеживать изменения PDF/Lua и обновлять проект

**Задачи:**

1. **Файловый мониторинг**
   - Использовать Java `WatchService` для отслеживания изменений
   - Обнаруживать изменения в PDF/Lua файлах

2. **Автоматическое обновление**
   - При изменении файла вызывать `update`
   - Логировать изменения

**Ожидаемый результат**: команда `watch` работает в фоне и автоматически обновляет проект

### Использование ИИ/LLM

**Роль ИИ в проекте:**

**Будущее (опционально):**

## Текущие ограничения и риски

**Ограничения:**
- PDF-парсинг сложен — не все PDF имеют структурированные таблицы
- Lua-анализ требует понимания доменной логики
- Инкрементальное обновление — сложная задача (конфликты с ручными правками)

**Риски:**
- Формат Monitor может быть не полностью документирован
- Разные проекты могут иметь разные структуры PDF/Lua
- Ручные правки инженеров могут конфликтовать с автоматической генерацией

**Митигация:**
- Начинаем с одного эталонного проекта ("Моцарелла")
- Тестируем на реальных данных
- Постепенно расширяем поддержку
